## Purpose

This document defines Jay’s operational flywheel across the ALMS work stack.

The flywheel does not claim authority. It describes movement: how an idea becomes an artifact, how an artifact becomes replayable, and how replayable work becomes durable memory.

## Core Flywheel

```text
IDEA
  -> ARTIFACT
  -> COMMIT
  -> HASH
  -> REPLAY
  -> ANCHOR
  -> PUBLISH
  -> WITNESS
  -> MEMORY
  -> NEXT IDEA
```

## ALMS Layer Map

```text
L0 — JOY
     artifact birth, canonical JSON, README, cultural packets, render staging

L1 — COMPUTERWISDOM
     deterministic replay, byte-for-byte checks, hash receipts, witness logic

L2 — AL
     anchoring prep, attestation machinery, ledger protocol experiments

L3 — External Surfaces
     IPFS, Base, EAS, ENS, Zora, public publication and addressing surfaces

L4 — Public Witness
     independent replay, outside confirmation, contradiction intake
```

## Work Stack

```text
WORK WISDOM
  -> choose the lane
  -> name the artifact
  -> define the boundary conditions

WORK FLOW
  -> create files
  -> stage evidence
  -> commit to repo
  -> compute hashes

WORK STACK
  -> JOY files it
  -> COMPUTERWISDOM checks it
  -> AL prepares it
  -> external surfaces publish or anchor it
  -> witnesses replay it
```

## Layer Rules

```text
JOY births.
COMPUTERWISDOM checks.
AL prepares anchors.
External surfaces publish or attest.
Witnesses replay.
```

No layer promotes itself.

No layer grants authority by existing.

Every layer must preserve the distinction between prepared, committed, replayed, anchored, published, and witnessed.

## Doctrine

```text
If it is not filed, it cannot be replayed.
If it is not replayed, it should not be anchored.
If it is not anchored, it should not pretend to be permanent.
If it is not witnessed, it should not claim public verification.
```
