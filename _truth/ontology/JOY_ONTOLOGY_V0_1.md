## Purpose

Define the noun layer of JOY.

JOY already verifies assets, receipts, manifests, oracle alignment, and replay harness state. This ontology defines the shared vocabulary future assets, rituals, receipts, courts, witnesses, and epochs must use.

This document creates no authority. It defines meanings for replay.

## Core Rule

Family First. Integrity Locked.

## Constitutional Strata

### Mechanics

The files, hashes, receipts, manifests, commits, and replay harnesses that allow JOY to verify state.

### Doctrine

The guiding commitments that prevent drift:

- no authority by narrative
- no canon without receipts
- no asset without hash
- no lineage without replay
- no economy above family

### Ontology

The nouns JOY uses to describe itself.

Ontology defines meaning, not power.

## Core Terms

### badge_family

A group of related badge assets sharing a common epoch, purpose, and verification pattern.

### badge_instance

A single badge asset represented by an SVG, receipt, manifest entry, and replay result.

### receipt

A JSON evidence object that binds a claim to a file hash, status, and replay context.

### ritual

A human-readable process that gives meaning to a mechanical verification path without replacing it.

### oracle

A known-good baseline used to compare future state against previously verified state.

### witness

A verifying surface that confirms whether local state, remote state, hash state, or replay state agrees.

### clean_era_artifact

An artifact born after recovery closed and verified without recovery scaffolding.

### meme_court_gate

The admissibility boundary requiring family_priority, economy_subordinate, meme_court_safe, and authority false.

### loom

The commit surface where threads are bound into replayable history.

### thread

A hash identity, usually SHA-256, representing the byte-level state of an asset.

### weave

The binding of asset, hash, receipt, manifest entry, ledger state, and remote witness.

### lineage

The ancestry chain that allows an artifact to be traced through receipts, commits, and replay outputs.

### epoch

A named era boundary in JOY history, such as recovery mode or post-recovery clean era.

## Badge Family Grammar

A badge family must declare:

- family identity
- epoch
- purpose
- member badge instances
- verification pattern
- admissibility gate

A badge instance must declare:

- asset_id
- asset_version
- asset_path
- receipt_path
- sha256_restored
- status
- replay result

## Legitimacy Doctrine

JOY does not rely on charisma, perfect archives, uninterrupted history, or memory alone.

JOY legitimacy comes from:

- replay
- reconstruction
- receipts
- remote witness
- lawful continuity repair

A recovered artifact can be legitimate when its recovery is visible, bounded, and replayable.

A clean-era artifact is legitimate when it is born inside verified build mode and passes the replay harness.

## Canonicalization Governance

An artifact becomes canon only when the required path is complete:

SVG → SHA-256 → Receipt → Manifest → Ledger → GitHub Commit → Remote Witness Replay

No artifact becomes canon by narrative alone.

## Ontology Membrane

### Canonical

Artifacts that are committed, replayable, and verified.

### Operational

Tools and scripts that help verify or manage canonical artifacts.

### Experimental

Drafts, branches, sketches, and tests that have not crossed the canon path.

### Forbidden

Claims of authority, custody, lineage, or verification without receipts and replay.
